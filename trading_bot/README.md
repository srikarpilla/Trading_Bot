# Binance Futures Testnet Trading Bot

A small, structured Python CLI application that places **MARKET**, **LIMIT**,
and **STOP (stop-limit, bonus)** orders on **Binance USDT-M Futures Testnet**,
with input validation, structured logging, and clean error handling.

## Project Structure

```
trading_bot/
  bot/
    __init__.py
    client.py          # Binance REST API wrapper (signing, requests, error mapping)
    orders.py           # Order construction, placement, response formatting
    validators.py        # Pure input validation functions
    exceptions.py         # ValidationError / NetworkError / BinanceAPIError
    logging_config.py      # Rotating file + console logging setup
  cli.py                    # argparse CLI entry point
  scripts/
    simulate_for_sample_logs.py  # generates the sample logs (see note below)
  tests/
    test_validators.py
    test_orders.py
    test_client.py
  sample_logs/
    sample_trading_bot.log        # example log output (see sample_logs/README.md)
  logs/                             # created at runtime, trading_bot.log lives here
  requirements.txt
  requirements-dev.txt
  .env.example
  README.md
```

**Why this layout:** `client.py` is the only module that knows how to talk
HTTP to Binance (signing, retries, error mapping). `orders.py` turns
validated CLI input into a Binance payload and formats the response —
it depends only on the client, not on argparse. `cli.py` is the thin
presentation layer. This separation is what makes `orders.py` and
`validators.py` fully unit-testable without any network access (see
`tests/`).

## Setup

### 1. Get Binance Futures Testnet credentials
1. Go to https://testnet.binancefuture.com and register/log in (GitHub login supported).
2. Once activated, generate an **API Key** and **Secret Key** from the testnet dashboard.
3. (Optional) Use the testnet faucet to top up your test USDT balance.

### 2. Clone / unzip and install dependencies
```bash
cd trading_bot
python3 -m venv .venv
source .venv/bin/activate        # Windows: .venv\Scripts\activate
pip install -r requirements.txt
```

### 3. Configure credentials
```bash
cp .env.example .env
# then edit .env and fill in:
#   BINANCE_TESTNET_API_KEY=...
#   BINANCE_TESTNET_API_SECRET=...
```

## Running the Bot

```bash
# Market order
python cli.py --symbol BTCUSDT --side BUY --type MARKET --quantity 0.01

# Limit order
python cli.py --symbol BTCUSDT --side SELL --type LIMIT --quantity 0.01 --price 65000

# Stop-limit order (bonus third order type)
python cli.py --symbol BTCUSDT --side BUY --type STOP --quantity 0.01 --price 64000 --stop-price 64100

# Validate input without sending anything to Binance
python cli.py --symbol BTCUSDT --side BUY --type MARKET --quantity 0.01 --dry-run
```

### CLI arguments

| Flag | Required | Notes |
|---|---|---|
| `--symbol` | yes | e.g. `BTCUSDT` (case-insensitive, normalized to upper) |
| `--side` | yes | `BUY` or `SELL` |
| `--type` | yes | `MARKET`, `LIMIT`, or `STOP` |
| `--quantity` | yes | must be a positive number |
| `--price` | for LIMIT/STOP | limit price |
| `--stop-price` | for STOP | trigger price for the stop-limit order |
| `--time-in-force` | no | `GTC` (default), `IOC`, or `FOK` |
| `--dry-run` | no | validate and print the request, skip the API call |

### Example output
```
Order Request Summary
----------------------
  Symbol      : BTCUSDT
  Side        : BUY
  Type        : MARKET
  Quantity    : 0.01

Order Response
--------------
  Order ID     : 5028953
  Symbol       : BTCUSDT
  Status       : FILLED

  Avg Price    : 65123.40

```

On failure, the console prints a short, specific message
(`Invalid input: ...`, `Order failed (Binance API error): ...`, `Order
failed (network error): ...`), while the full technical detail
(including tracebacks) goes to `logs/trading_bot.log`.

## Logging

Every request and response is logged to `logs/trading_bot.log`
(rotating, 2 MB per file, 5 backups):
- `DEBUG` — outgoing request params (signature stripped) and raw response body
- `INFO` — order lifecycle milestones (placing order, order placed, orderId/status)
- `ERROR` — validation failures, Binance API errors, and network failures (with traceback)

Console output stays clean (just the summary/response/success-or-failure
message); the file carries the full audit trail.

## Testing

The test suite covers validators, order construction, order placement
(with a mocked client), and the client's HTTP/signing layer (with a
mocked `requests.Session`) — no real network or API keys required:

```bash
pip install -r requirements-dev.txt
pytest tests/ -v
```

## About `sample_logs/`

This assignment asks for log files from at least one MARKET and one
LIMIT order. `sample_logs/sample_trading_bot.log` was generated by
running the **real application code** end-to-end
(`cli.py` → `bot/orders.py` → `bot/client.py` → `bot/logging_config.py`)
with only the raw HTTP transport stubbed out, because the environment
this was built in has no outbound network access to
`testnet.binancefuture.com`. See `sample_logs/README.md` and
`scripts/simulate_for_sample_logs.py` for exactly what was stubbed and
why. When you run this project with your own testnet API keys against
the live endpoint, `logs/trading_bot.log` will contain genuine
request/response entries in this same format — that's the file to
submit as your log evidence.

## Assumptions

- Binance Futures Testnet USDT-M REST API is accessed directly via
  `requests` (no `python-binance` dependency), so the client layer is
  transparent and easy to review/extend.
- `STOP` is implemented as Binance's `STOP` order type (stop-**limit**:
  requires both `price` and `stopPrice`), chosen as the bonus third
  order type over OCO/TWAP/Grid for simplicity within the ~60 minute
  scope.
- Quantities/prices are treated as `Decimal` internally for precision,
  then serialized to strings before being sent to Binance (as the API
  expects).
- No symbol-existence check against live `exchangeInfo` is made before
  submitting an order (would add a network round-trip); Binance itself
  will reject an unknown symbol, and that error is surfaced cleanly via
  `BinanceAPIError`.
- `recvWindow` defaults to 5000 ms; timestamps are generated
  client-side (Binance testnet performs its own clock-skew checks).
